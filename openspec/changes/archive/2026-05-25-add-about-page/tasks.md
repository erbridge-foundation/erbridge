## 1. Backend: build-time commit SHA capture

- [x] 1.1 Add `backend/build.rs` that runs `git rev-parse --short HEAD`, captures stdout, and emits `cargo:rustc-env=GIT_COMMIT_SHA=<sha>`. If the command fails for any reason (no `git` on PATH, no `.git/` directory, non-zero exit), the script SHALL emit `cargo:rustc-env=GIT_COMMIT_SHA=unknown` and exit 0. The script SHALL also emit `cargo:rerun-if-changed=.git/HEAD` and `cargo:rerun-if-changed=.git/refs/` so the SHA is re-baked when the working tree's HEAD moves.
- [x] 1.2 Add a unit test (in `backend/build.rs` is not testable directly; add the test to `backend/src/handlers/health.rs` instead): assert `env!("GIT_COMMIT_SHA")` resolves to a non-empty string at compile time (this fails to compile if `build.rs` did not run).
- [x] 1.3 Verify locally: `cargo build` from inside the repo with `.git/` present produces a binary whose health handler reports a real SHA; `cargo build` from a tarball-extracted copy without `.git/` produces a binary that reports `"unknown"`. Document the verification method in the commit message.
- [x] 1.4 Update `backend/Dockerfile` to copy `build.rs` into the builder stage (add it to the existing `COPY Cargo.toml Cargo.lock ... ./` line). Without this, Cargo does not run the build script in the image and `env!("GIT_COMMIT_SHA")` fails to compile (`environment variable GIT_COMMIT_SHA not defined at compile time`). The Docker build has no `.git/` in its context, so the script resolves `commit` to `"unknown"` (design §3) — acceptable. `build.rs` also honours an explicit `GIT_COMMIT_SHA` build arg if a deployment later wants a real SHA in the image. Verified by `docker compose -f docker-compose.dev.yml --env-file .env.dev build backend` succeeding and `GET /api/health` returning `"commit": "unknown"`.

## 2. Backend: /api/health endpoint

- [x] 2.1 Add `backend/src/handlers/health.rs` per the `rust-rest-api` skill's handler-layer rules. Define a `HealthResponse` struct (in `backend/src/dto/health.rs` per the skill's DTO layering rule) with `#[derive(Serialize, utoipa::ToSchema)]` and fields `status: HealthStatus`, `version: String`, `commit: String`, `components: Vec<ComponentHealth>`. Define `HealthStatus` and `ComponentStatus` as serde-string-enum types (`#[serde(rename_all = "snake_case")]`) with variants `Ok` and `Degraded`. Define `ComponentHealth { name: String, status: ComponentStatus }`.
- [x] 2.2 Implement the service-layer function `backend/src/services/health.rs::check(pool: &PgPool) -> HealthSnapshot` that runs `sqlx::query!("SELECT 1 AS one").fetch_one(pool).await` and returns a `HealthSnapshot { db: ComponentStatus }`. Map `Ok(_)` to `Ok`, `Err(_)` to `Degraded`. Do NOT propagate the error — degraded is the success signal, the endpoint returns 200 in both cases.
- [x] 2.3 Implement the handler function `backend/src/handlers/health.rs::get_health(State(state): State<AppState>) -> Json<HealthResponse>` that calls `services::health::check(&state.pool).await`, builds the `components` array (`[{ name: "db", status: snapshot.db }]`), derives the overall status (`Ok` iff every component is `Ok`, else `Degraded`), and returns `Json(HealthResponse { status, version: env!("CARGO_PKG_VERSION").into(), commit: env!("GIT_COMMIT_SHA").into(), components })`. The handler signature MUST NOT name the `AuthenticatedAccount` extractor (that is what makes the route public) and the body MUST NOT consult the session store, API keys, or any auth machinery.
- [x] 2.4 Add a unit test in `backend/src/services/health.rs` for the overall-status aggregation rule: pass a snapshot with all-`Ok` components and assert the overall is `Ok`; pass one with at least one `Degraded` and assert the overall is `Degraded`. (Pure-function test; no DB needed.)
- [x] 2.5 Register the route in `backend/src/lib.rs::build_router` (NOT `main.rs` — `main.rs` only calls `build_router`): add `.route("/api/health", get(handlers::health::get_health))` next to the `SwaggerUi` merge that registers `/api/openapi.json` and `/api/docs`. There is **no** public/authenticated router split and **no** `AuthenticatedAccount` middleware layer in this codebase — auth is per-handler via the `AuthenticatedAccount` extractor. The route is public precisely because `get_health` does NOT name that extractor in its signature (per §2.3). Add a one-line comment that this is the documented carve-out from the `api-contract` envelope rule. Health is NOT a versioned route — do not nest it under `/api/v1`, and do not add it to `registered_api_v1_routes()`.
- [x] 2.6 Annotate the handler with `#[utoipa::path(get, path = "/api/health", responses((status = 200, description = "Health snapshot", body = HealthResponse)))]`. Add `HealthResponse`, `HealthStatus`, `ComponentHealth`, and `ComponentStatus` to the `#[derive(utoipa::OpenApi)]` collector in `backend/src/openapi.rs` so they appear in `/api/openapi.json`. Confirm via `curl $APP_URL/api/openapi.json | jq '.paths."/api/health"'` that the route is documented and its response schema is the flat `HealthResponse` (NOT wrapped in `ApiResponse<T>`).
- [x] 2.7 Add `backend/tests/health.rs` (integration test using `#[sqlx::test]` per the `rust-rest-api` skill): build the router, hit `GET /api/health`, assert HTTP 200, parse the body as `HealthResponse`, assert `version` is non-empty, `commit` is non-empty, `components` contains exactly one element with `name = "db"` and `status = Ok`, and the overall `status = Ok`.
- [x] 2.8 Add `backend/tests/hurl/health.hurl` per the `rust-rest-api` skill's HURL convention. Assertions: `HTTP 200`; `jsonpath "$.status" == "ok"`; `jsonpath "$.version" exists`; `jsonpath "$.commit" exists`; `jsonpath "$.components" count == 1`; `jsonpath "$.components[0].name" == "db"`; `jsonpath "$.components[0].status" == "ok"`; **and** `jsonpath "$.data" not exists` (proves the response is NOT enveloped).
- [x] 2.9 Add a strict-drift case for `/api/health` in `backend/tests/openapi_strict.rs` (foundation §2d.6): a `#[sqlx::test]` that hits `GET /api/health` and calls `assert_schema(&doc, "/api/health", "get", "200", &body)`, validating the live 200 response against the flat `HealthResponse` schema declared by the annotation. (The existing tests are one function per route, not a `cases` list — follow that shape; e.g. mirror `get_me_200_matches_schema`.) No special-casing needed — `assert_schema` validates against whatever schema the `#[utoipa::path]` annotation declares.
- [x] 2.10 Add a **fail-closed auth-coverage** test to `backend/tests/openapi_strict.rs` (sibling to `all_registered_routes_are_documented`, design §10 / api-health spec): iterate `backend::registered_api_v1_routes()`, and for each `(path, method)` assert the OpenAPI document's operation at `/paths/<path>/<method>/security` exists and is a non-empty array. Reuse the existing `openapi_doc_json()` helper and the same `~1` path-escaping used in `compile_schema`. Fail with a message naming the offending route. This guards against a future `/api/v1/*` handler silently going public by omitting the `AuthenticatedAccount` extractor. `/api/health` is not in `registered_api_v1_routes()`, so it is correctly out of scope. Verify the test bites: temporarily strip the `security(...)` clause from one v1 handler's `#[utoipa::path]`, confirm the test fails, then restore it.
